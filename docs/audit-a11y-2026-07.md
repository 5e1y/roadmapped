## 1. Clavier (#107)

**Verdict global : sain.** Quasi tous les interactifs sont de vrais `<button>`, zéro
`tabIndex` positif, et le `:focus-visible` global d'`index.css` (hors `@layer`, donc il bat
les utilitaires Tailwind) donne un focus visible presque partout. Le SidePanel gère Esc en
cascade, mémorise le déclencheur et restaure le focus. **Les croix de tags des combobox sont
conformes** : c'est du Base UI `Combobox.ChipRemove` (`tabIndex=-1` par design, avec
équivalent clavier natif ←/→ + Backspace depuis l'input, refocus automatique, `aria-label`
« Retirer X » — vérifié dans le source de @base-ui/react 1.6.0). Aucun bloquant.

Le motif dominant des findings : **le focus perdu APRÈS une action** (retombe sur
`document.body`).

| Élément | Localisation | Problème | Sév. | Fix |
|---|---|---|---|---|
| Croix ✕ suppression de note | NotepadView.tsx:144-147 | Focusable mais invisible au focus (`opacity-0` levé par `group-hover` seulement) — action destructive sans confirmation | gênant | `focus-visible:opacity-100` (pattern RemoveButton TaskPanel.tsx:132) + confirmation |
| AddCombobox (deps/liens) | ui.tsx:215-222 | Remontage `key={epoch}` après ajout → l'input focalisé est démonté, focus sur body | gênant | Refocus du nouvel input (ref + useEffect sur epoch) ou vidage contrôlé |
| `blurOnEnter` des champs ghost | ui.tsx:26-28 | Enter = `blur()` → focus sur body, Tab repart du haut du document | gênant | save() sans blur, ou refocus après |
| Lignes supprimables (RemoveButton) | TaskPanel.tsx:125-137 | La ligne (et le bouton focalisé) disparaît après reload — pas de repositionnement | gênant | Focus sur ligne suivante / combobox de section |

Collapsible, Input = Base UI (a11y incluse). Faits main et acceptables : SidePanel (dialog
non modal, Esc + restauration ✓), tabs ViewHeader (`aria-current` ✓), toggles
(`aria-pressed` ✓), zoom graphe (`aria-label` ✓). Aucun menu/dialog artisanal inaccessible.

## 2. Contraste WCAG AA (#108)

**Verdict global : problème structurel massif.** L'encre principale et l'accent passent
largement (#171717 = 17.9:1, accent #2563eb = 5.17:1). Mais **`text-neutral-400`
(#a1a1a1, ~86 occurrences — la couleur de texte la plus utilisée du dashboard) ne fait que
2.58:1** sur blanc (seuil : 4.5:1), et toute la strate `neutral-300` (#d4d4d4, 1.48:1) est
quasi invisible, y compris sur des contrôles interactifs (seuil composants : 3:1).

**Le fix systémique (retenu pour design.md #111)** : promotion générale
`neutral-400 → neutral-500` (#737373 = 4.74:1 sur blanc, 4.54:1 sur #fafafa) et
`neutral-300 → neutral-500` pour tout ce qui est interactif ou porteur d'information ;
`neutral-500 → neutral-600` quand le fond est neutral-100/200.

## Synthèse pour design.md (#111)

1. **Le tri-couche est déjà écrit** (index.css:23 : page #fafafa / carte #fff / filets
   #e5e5e5) — il faut l'appliquer aux 2 vues qui le violent, pas l'inventer.
2. **La décision de contraste est systémique, pas au cas par cas** : plancher
   `neutral-500` pour tout texte/contrôle porteur de sens ; `neutral-600` sur fond gris ;
   neutral-300/400 réservés au purement décoratif.
3. **Un seul îlot non-Base UI** (select natif MiniZone) — le remplacer clôt la question
   « passe BaseUI » ; le reste est déjà du Base UI ou du fait-main acceptable.
4. **Le langage « actif » existe** (accent-tint + filet) — DocsTree est le seul déviant.
5. Sortir l'ambre/rouge du Notepad OU décider un registre destructif global.
6. Le motif a11y n° 1 à traiter en #115 : la gestion du focus après action.
