# Vue Graphe v2 — layout par lib, rendu SVG conservé

**Date** : 2026-07-08 · **Statut** : DRAFT — en attente de relecture Rémi
**Contexte** : `2026-07-07-roadmaped-v2-design.md` (§ Vue Roadmap → mode Graphe achievement)
**Touche** : `src/components/RoadmapGraph.tsx`, `src/lib/roadmap.ts`

## Problème

Le mode Graphe actuel (`RoadmapGraph.tsx`) place les nœuds à la main :

- **colonnes = sections** (`colOf`, `xOf`), **rangées = couche topologique** (`topoLayers`
  + un plancher `nextRow` pour éviter les collisions verticales). Grille rigide.
- Conséquence directe : dès qu'une dépendance **traverse plusieurs sections** ou saute
  plusieurs rangées, l'arête part en diagonale de gouttière (`gutterX`) et **croise**
  d'autres arêtes et d'autres cartes. Rien ne minimise les croisements — la position
  d'un nœud ne dépend que de sa section et de sa profondeur, jamais de ses voisins.
- L'axe horizontal encode la **section**, pas le **flux de dépendances**. On ne lit donc
  pas « ce qui débloque quoi » : deux tâches enchaînées peuvent être côte à côte dans la
  même colonne (empilées) ou à l'opposé de l'écran.
- Le zoom est un simple `scale` (`useState`) posé sur une boîte, le pan est le scroll
  natif du conteneur. Pas de zoom-vers-le-curseur, pas de drag pour naviguer.
- Aucune lecture de chemin : survoler un nœud n'éclaire pas ses prérequis amont ni ses
  dépendants aval. Sur 50+ tâches, l'écheveau devient illisible.

La mécanique de sélection des nœuds, elle, est **saine et à garder** : filtrage des
`quick`, masquage des `done` sauf s'ils sont prérequis transitifs d'un ticket visible
(la boucle `keep`/`grew`), états `computeAvailability` / `missingPrereqs`, séparation
prérequis-dans-le-graphe vs hors-graphe (`missingHidden`).

## Décision (Rémi)

- **Le placement des nœuds passe à une lib de layout de graphe.** Objectif : minimiser
  les croisements, ranger proprement selon le flux de dépendances. On arrête la grille
  section×couche faite main.
- **Lib retenue : `dagre`** (le fork maintenu `@dagrejs/dagre`). Voir comparatif ci-dessous.
- **On CONSERVE le rendu SVG monochrome existant** : `defs`/`marker` `rm-arrow`, arêtes
  `stroke #737373` pointillées `3 3`, `GraphCard` et son langage d'états inchangés. La lib
  ne fournit **que des coordonnées** ; elle ne rend rien, n'ajoute aucun DOM, aucun style.
- **Zoom/pan = petit hook maison** (`useZoomPan`), pas de dépendance.
- **PAS de React Flow** (rendu propriétaire, DOM imposé, poids, incompatible avec notre
  SVG monochrome).

### dagre vs elkjs — pourquoi dagre

| | **dagre** (`@dagrejs/dagre`) | **elkjs** |
|---|---|---|
| Poids | ~90 KB min / **~28 KB gzip** (graphlib inclus) | ~1,5 Mo min / **~500 KB gzip** (portage GWT de Java) |
| Exécution | synchrone, <50 ms pour 50–100 nœuds | conseillé en **web worker** (async, sinon jank) |
| Algorithme | Sugiyama en couches (rang → ordre médian → coord), DAG | idem + routage orthogonal natif, ports, partitions |
| Groupes | **nœuds composés** (`setParent`) + rectangles de cluster | idem, plus riche |
| API | `new dagre.graphlib.Graph()` → `dagre.layout(g)`, trivial | JSON verbeux, cycle de vie worker |
| Types | fournis par le paquet | fournis |

Pour un DAG de tickets à l'échelle 50–200 nœuds, **elkjs est surdimensionné** : +470 KB
gzip et un worker pour une qualité de layout marginale ici. dagre fait exactement le
travail (couches + tri des croisements + clusters), reste synchrone, et son poids est
négligeable devant `marked` déjà embarqué. **On prend dagre.**

> Dépendance npm : **`@dagrejs/dagre`** (le paquet `dagre` d'origine est en déshérence ;
> le fork `@dagrejs` est la version maintenue, types TS inclus, pas de `@types/*` à part).
> Poids ajouté au bundle : **~28 KB gzip**.

## Conception

Tout le calcul de positions quitte `RoadmapGraph.tsx` pour `src/lib/roadmap.ts` (testable,
pur, mémoïsable). Le composant ne fait plus que : sélectionner les nœuds (logique
existante), appeler l'adaptateur, dessiner.

### 1. Adaptateur de layout (`roadmap.ts`)

Entrée = les nœuds déjà filtrés + leurs arêtes internes + leur section. Sortie = positions
absolues, boîtes de section, et les points d'arête. dagre est un **détail d'implémentation
caché derrière cette fonction** — le composant n'importe jamais dagre.

```ts
export interface GraphInput {
  nodes: { id: number; sectionKey: string }[]   // déjà filtrés (quick exclus, done masqués)
  edges: { from: number; to: number }[]          // from = prérequis, to = dépendant (dans-le-graphe)
  sectionOrder: string[]                         // ordre NN des sections (rang de partition)
}
export interface GraphLayout {
  nodes: Map<number, { x: number; y: number; w: number; h: number }>  // coin haut-gauche
  clusters: Map<string, { x: number; y: number; w: number; h: number; title: string }>
  edges: Map<string, { points: { x: number; y: number }[] }>           // clé `${from}->${to}`
  width: number; height: number
}
export function graphLayout(input: GraphInput): GraphLayout
```

Corps :

- `const g = new dagre.graphlib.Graph({ compound: true })`
- `g.setGraph({ rankdir: 'LR', nodesep: 24, ranksep: 64, marginx: PAD, marginy: PAD })`
  — **`rankdir: 'LR'`** : le prérequis est à gauche, le dépendant à droite. L'axe X
  encode enfin le **flux de dépendances** (ce qu'on veut lire), plus la section.
- **Regroupement par section = nœuds composés.** Un nœud-cluster par section
  (`g.setNode('sec:'+key, { label })`), et chaque tâche `g.setParent(id, 'sec:'+key)`.
  dagre garde les membres d'une section **groupés** et calcule le rectangle englobant du
  cluster (`g.node('sec:'+key)` → `{x,y,width,height}`, centre). On en tire `clusters`.
  Le groupement est ainsi **natif du layout** (pas un post-traitement qui pourrait
  chevaucher), tout en laissant dagre minimiser les croisements globalement.
- Fixer la taille de chaque nœud à `CARD_W × CARD_H` (constantes conservées) avant layout.
- `dagre.layout(g)`. dagre donne des **centres** ; on convertit en coin haut-gauche
  (`x - w/2`, `y - h/2`) pour coller au positionnement absolu actuel de `GraphCard`.
- `edges` : on récupère `g.edge(e).points` (spline déjà routée autour des rangs par dagre).

**Mémoïsation** : même pattern que `availabilityCache` (WeakMap par identité de `tree`).
Ici la clé combine `tree` **et** l'état `showDone` (le jeu de nœuds change) → `Map` à deux
niveaux `WeakMap<TaskTree, Map<boolean, GraphLayout>>`. Le layout n'est recalculé qu'à
une écriture (nouveau `tree`) ou un toggle `showDone`, pas à chaque render/hover/zoom.

### 2. Routage des arêtes lisible

On garde le **rendu** SVG actuel (path pointillé + `marker rm-arrow`), on change la
**géométrie** : au lieu des coudes de gouttière faits main, on trace la polyligne
`edges.points` fournie par dagre, **arrondie aux coins** pour la lisibilité.

- Path : `M` sur le premier point, segments successifs, chaque sommet interne adouci par
  un petit quart de courbe (`Q`) de rayon fixe (~8 px) — coins nets mais pas anguleux,
  cohérent avec l'esthétique « filets » du design.md.
- `markerEnd="url(#rm-arrow)"` inchangé : la flèche pointe le dépendant (aval).
- Les arêtes restent **derrière** les cartes (`svg` en premier, `pointer-events-none`).
- Le cas « prérequis hors-graphe » ne trace toujours **aucune** arête (déjà géré : on ne
  pousse une arête que si `nodeIds.has(depId)`), et reste signalé en texte dans la carte
  (`missingHidden`).

### 3. Zoom / pan — hook maison `useZoomPan`

Nouveau `src/components/useZoomPan.ts` (ou `state/`). Remplace le `scale`+scroll actuel.

```ts
function useZoomPan(opts?: { min?: number; max?: number })
  : { scale; tx; ty;
      onWheel; onPointerDown;               // handlers à poser sur le viewport
      fit(contentW, contentH, viewport);    // « Ajuster »
      reset(); }
```

- État `{ scale, tx, ty }`. Transform appliqué sur la boîte de layout :
  `translate(${tx}px, ${ty}px) scale(${scale})`, `transform-origin: 0 0`.
- **Molette = zoom vers le curseur** : on ajuste `tx/ty` pour que le point sous la souris
  reste fixe (formule d'ancrage classique). `clamp(scale, min=0.2, max=2.5)`.
- **Drag = pan** : `onPointerDown` sur le fond (pas sur une carte) → `setPointerCapture`,
  `pointermove` incrémente `tx/ty`. Curseur `grab`/`grabbing`.
- **`fit`** rejoue la logique de `fitWidth` actuelle mais sur les deux axes (scale =
  min(vpW/contentW, vpH/contentH), centré). Le bouton « Ajuster » existant l'appelle ;
  on ajoute « Réinitialiser » (100 %).
- Le conteneur passe en `overflow-hidden` (le pan remplace le scroll natif). Les boutons
  de zoom épinglés (`absolute right-3 top-3`) et leurs `aria-label` sont conservés ;
  `+`/`−` appellent un `zoomBy` centré sur le milieu du viewport.
- **A11y** : le viewport est `tabIndex=0`, flèches = pan, `+`/`−` = zoom (mineur, aligné
  sur la politique a11y du repo).

### 4. Surlignage du chemin amont / aval

Nouveau helper pur dans `roadmap.ts`, à côté de `reverseDependents` :

```ts
/** Fermeture transitive d'un nœud dans le sous-graphe affiché :
 *  ancestors = prérequis (amont), descendants = dépendants (aval). */
export function graphNeighborhood(
  edges: { from: number; to: number }[], id: number
): { ancestors: Set<number>; descendants: Set<number> }
```

- Deux BFS sur les listes d'adjacence (amont via `to→from`, aval via `from→to`), bornées
  au sous-graphe **affiché** (mêmes arêtes que le rendu). O(V+E), trivial à 50–200 nœuds.
- État composant : `const [focus, setFocus] = useState<{ id, sticky } | null>()`.
  - **Survol** d'une carte → `focus` transitoire (`sticky:false`).
  - **Clic** sur le fond de carte (pas le bouton d'ouverture du panneau) → bascule
    `sticky:true` ; re-clic sur le fond / `Esc` / clic vide → efface. Le clic sur la carte
    ouvre toujours le panneau (`openTask`) — le surlignage sticky est un modificateur
    (ex. `alt`+clic, ou zone dédiée) à trancher (voir points ouverts).
- Rendu du surlignage (monochrome, pas de couleur nouvelle) :
  - hors chemin → cartes et arêtes atténuées (`opacity`/encre `neutral-300`, jamais
    d'opacity sur le conteneur — rappel du commentaire existant : sinon les arêtes
    transparaissent) ;
  - nœud focalisé → contour `neutral-900` ;
  - arêtes du chemin → `stroke #171717`, trait plein (on lève le `dasharray`), les autres
    restent `#737373` pointillées et atténuées.
  - `set.has(id)` décide par carte/arête ; aucun recalcul de layout (le focus ne touche
    que des classes, le layout mémoïsé est intact).

### 5. Regroupement visuel par section

Les rectangles `clusters` de l'adaptateur remplacent les bandes de colonnes actuelles
(`sections.map(... border-l ...)`). Pour chaque cluster : un rectangle au filet
`neutral-200`, coins arrondis, fond très léger, **derrière les arêtes**, avec le titre de
section en `text-xs font-semibold` repris tel quel (encre `neutral-900` peuplé /
`neutral-300` vide — hiérarchie d'encre conservée du code actuel). Les sections
`abandoned` restent exclues (filtre existant).

### 6. État visuel des milestones bloquants — dépend de #6, référencé sans bloquer

La **sémantique** des milestones bloquants (quel champ, portée du verrou : tout le
backlog ? les sections suivantes ?) est spécifiée par la tâche **#6**
(`docs/specs/AAAA-MM-JJ-progress-milestones.md`, non encore approuvée). La Vue Graphe v2
**ne décide pas** cette sémantique et **ne se bloque pas dessus** :

- `GraphCard` reçoit son état via une **fonction injectée** `stateOf(task): Availability |
  'milestone-blocked'` plutôt que de lire `avail` en direct. Tant que #6 n'est pas livrée,
  `stateOf` renvoie exactement `computeAvailability` (aucun changement de comportement).
- On **réserve** dès maintenant le 5e cas visuel `milestone-blocked` dans le `skin` de
  `GraphCard` (traitement plus fort que `locked` : filet `neutral-900` + hachure légère,
  toujours monochrome) et un badge « Jalon bloquant ». Ce cas est **inatteignable** avant
  #6 — c'est un point d'ancrage, pas une implémentation de la règle.
- Quand #6 atterrit, seule `stateOf` change (dans `roadmap.ts`), pas le rendu du graphe.

### Tenue à 50+ tâches

- Layout dagre synchrone <50 ms à cette échelle, **mémoïsé** (§1) → payé une fois par
  écriture, jamais au hover/zoom/pan.
- Rendu = SVG + divs absolus (comme aujourd'hui), pas de virtualisation nécessaire avant
  quelques centaines de nœuds. Si un jour ça grince : c'est le pan/zoom qui protège
  (moins de DOM visible), la virtualisation reste hors périmètre (YAGNI).
- Le surlignage (§4) ne change que des classes CSS, pas le DOM ni le layout.

### Ce qui disparaît de `RoadmapGraph.tsx`

`COL_W, COL_GAP, ROW_H, CARD_W(?), PAD, HEADER_H` (les gabarits de grille), `xOf`, `yOf`,
le `colOf`, la boucle de placement `nextRow`/`layerOf`, le routage `gutterX`, le
`scale`+scroll. `CARD_W`/`CARD_H` sont conservés (taille de nœud passée à dagre).
`topoLayers` n'est plus utilisé par le graphe ; on le garde exporté tant qu'un autre
appelant existe (sinon suppression dans la tâche de nettoyage). La logique de sélection
des nœuds (§Problème) est **déplacée telle quelle** vers la construction du `GraphInput`.

## Découpage en tâches d'implémentation

Chaînables par `dependsOn` (ids à allouer par le CLI) :

1. **Dépendance + adaptateur `graphLayout`** (`roadmap.ts`) : ajouter `@dagrejs/dagre`,
   écrire `GraphInput`/`GraphLayout`/`graphLayout` (rankdir LR, clusters composés,
   conversion centre→coin, points d'arête), + mémoïsation WeakMap. Tests unitaires sur un
   petit DAG (positions cohérentes, clusters non vides, arêtes présentes). — *socle*
2. **Hook `useZoomPan`** : zoom-vers-curseur, drag-pan, `fit`, `reset`, clamp. Isolé,
   testable à part. — *indépendant de 1*
3. **Réécriture de `RoadmapGraph.tsx`** : brancher §1 (layout) + §2 (arêtes arrondies) +
   §3 (hook), supprimer la grille faite main, rendre les `clusters` (§5). *dependsOn: 1, 2*
4. **Surlignage amont/aval** : `graphNeighborhood` (`roadmap.ts` + test), état `focus`,
   atténuation monochrome, sticky/Esc. *dependsOn: 3*
5. **Point d'ancrage milestones bloquants** : indirection `stateOf`, 5e cas visuel
   `milestone-blocked` (inatteignable), badge. *dependsOn: 3* — **ne pas** implémenter la
   règle (c'est #6).
6. **Nettoyage + vérif 50+** : retirer le code mort (`topoLayers` si orphelin, constantes),
   jeu d'essai à 50+ tâches, contrôle perf (layout mémoïsé, pas de recalcul au hover). Vérif
   artefact : ouvrir la Vue Graphe sur les vraies données, zoomer/paner, survoler un nœud.
   *dependsOn: 4, 5*

## Risques / points ouverts

- **Clusters dagre** : le mode composé (`setParent`) peut, sur certains graphes,
  augmenter les croisements ou mal border les arêtes inter-clusters (bug historique dagre).
  **Repli** si les rectangles se chevauchent ou routent mal : abandonner le compound,
  grouper visuellement en post-layout (boîte englobant les nœuds d'une section) et/ou
  contraindre par `rank` — à valider sur les vraies données en tâche 3.
- **Arêtes rebroussantes** : sur un DAG large et LR, des arêtes de long saut peuvent
  repasser derrière des cartes. À surveiller ; le surlignage (§4) mitige la lecture. Si
  gênant, on pourra router via les points dagre sans arrondi, ou augmenter `ranksep`.
- **Perf réelle >150 nœuds** : non mesurée ici ; le repo n'est pas près d'y arriver.
  Virtualisation explicitement différée.

> **Décision verrouillée (Rémi, 2026-07-08)** : surlignage amont/aval = **survol seul
> (transitoire)**, pas de sticky ; le clic reste réservé à l'ouverture du panneau
> (`openTask`). Simplicité, aucun conflit de geste.
