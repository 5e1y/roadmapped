**Touche** : `index.html` (script anti-flash), `src/index.css` (tokenisation + bloc dark),

**Ne touche PAS** : `tempColor()` (la rampe coolwarm est inchangée), `BirdMascot`/`birdFrames`,

**Ticket** : #269.

## Principe

Tailwind v4 génère les utilitaires en `var()` (`.bg-white { background-color:
var(--color-white) }` — vérifié dans le CSS buildé). Donc **redéfinir les variables du bloc
`@theme` sous `:root[data-theme="dark"]` retourne toute l'app sans toucher un seul
composant**. Le mode sombre est un JEU DE VALEURS, pas un thème parallèle : zéro
`dark:` variant, zéro classe conditionnelle. Corollaire strict : **tout hex en dur est un
bug visuel en sombre** — la spec liste chacun et son remplacement (§4).

Deux faits vérifiés sur le build qui fondent la mécanique :

1. Le bloc `@theme` est émis dans `@layer theme` ; notre bloc d'override est du CSS
   **non-layered** dans `index.css` → il gagne toujours sur les défauts, quel que soit
   l'ordre. Le bloc dark est un règle CSS ordinaire (PAS dans `@theme`), placée juste
   après le `@theme` existant.
2. Les ombres sont INLINÉES par Tailwind avec la couleur en
   `var(--tw-shadow-color, #0000001a)`, et `--tw-shadow-color` est enregistrée
   `@property { inherits: false }` → un override sur `:root` ne se propage PAS. L'override
   d'ombre doit viser l'élément porteur de la classe (`:root[data-theme="dark"] .shadow-lg
   { --tw-shadow-color: … }`) — voir §6.

Les modificateurs d'opacité (`bg-accent/5`…) compilent en `color-mix(in oklab,
var(--color-accent) …)` : ils suivent le token au runtime, rien à faire.

## 1. Le tri-couche en sombre (pas un invert naïf)

Le modèle clair « page < carte < filets » est conservé, mais **la carte devient le
niveau clé** : elle prend `#171717` — l'encre du clair devient la surface du sombre, la
symétrie est voulue. La page passe SOUS la carte (l'élévation reste lisible : ce qui est
« posé » est plus clair que le sol). Pas de noir pur : le plancher est `≈#0f0f0f`, l'encre
plafonne à `#f5f5f5` (jamais `#fff` — pas de halo).

| Couche | Clair | Sombre |
|---|---|---|
| page (body) | `#fafafa` | `oklch(0.168 0 0)` ≈ `#0f0f0f` |
| carte (`bg-white`) | `#ffffff` | `oklch(0.205 0 0)` = `#171717` |
| filets (`neutral-200`) | `#e5e5e5` | `oklch(0.31 0 0)` ≈ `#303030` |
| encre (`neutral-900`) | `#171717` | `oklch(0.97 0 0)` = `#f5f5f5` |

Nouveau token **`--color-page`** (le fond du body est aujourd'hui un hex en dur) : ajouté
au `@theme` clair avec `#fafafa`, redéfini en sombre. Le body passe à
`background-color: var(--color-page); color: var(--color-neutral-900);`. La page ne peut
PAS réutiliser `neutral-50` en sombre : `hover:bg-neutral-50` sert de survol de rangée SUR
carte et doit donc être PLUS CLAIR que la carte, alors que la page est plus sombre.
D'où la scission page/50 (inexistante en clair où les deux valent ~`#fafafa`).

## 2. L'échelle neutral complète (clair | sombre)

Chaque échelon est pensé par son RÔLE dans le code, pas par symétrie mécanique. Le cœur
de l'échelle est un miroir exact des valeurs Tailwind canoniques (zéro dérive : les
oklch du clair resservent en face) ; seuls trois échelons (50, 200, 600) reçoivent une
valeur propre parce que leur rôle l'exige.

| Token | Clair | Sombre | Rôle observé dans le code |
|---|---|---|---|
| `--color-white` | `#fff` | `oklch(0.205 0 0)` `#171717` | carte, popovers, `text-white` inversé |
| `--color-page` (NOUVEAU) | `#fafafa` | `oklch(0.168 0 0)` ≈`#0f0f0f` | fond du body, uniquement |
| `--color-neutral-50` | `oklch(0.985 0 0)` | `oklch(0.24 0 0)` ≈`#1f1f1f` | `hover:bg-neutral-50` des rangées |
| `--color-neutral-100` | `oklch(0.97 0 0)` | `oklch(0.269 0 0)` `#262626` | hover des contrôles, fond code `.doc-prose` |
| `--color-neutral-200` | `oklch(0.922 0 0)` | `oklch(0.31 0 0)` ≈`#303030` | LES filets (bordures non interactives) |
| `--color-neutral-300` | `oklch(0.87 0 0)` | `oklch(0.371 0 0)` `#404040` | bordures des contrôles header, disabled bg |
| `--color-neutral-400` | `oklch(0.708 0 0)` | `oklch(0.556 0 0)` `#737373` | décoratif pur (× du header, disabled) |
| `--color-neutral-500` | `oklch(0.556 0 0)` | `oklch(0.708 0 0)` `#a3a3a3` | PLANCHER texte/contrôles porteurs de sens |
| `--color-neutral-600` | `oklch(0.439 0 0)` | `oklch(0.78 0 0)` ≈`#b9b9b9` | plancher sur fond gris (100/200) |
| `--color-neutral-700` | `oklch(0.371 0 0)` | `oklch(0.87 0 0)` `#d4d4d4` | texte secondaire appuyé (repo du header…) |
| `--color-neutral-800` | `oklch(0.269 0 0)` | `oklch(0.922 0 0)` `#e5e5e5` | quasi-encre (hover d'encre) |
| `--color-neutral-900` | `oklch(0.205 0 0)` | `oklch(0.97 0 0)` `#f5f5f5` | L'ENCRE |
| `--color-accent` | `#2563eb` | `#3b82f6` | actif / in_progress / attention (§3) |
| `--color-accent-tint` | `#eef3fd` | `#1c2636` | fond de sélection OPAQUE (§3) |

Effets automatiques à connaître (aucun code à changer, mais à vérifier à l'œil) :

- **Nav active / bouton primaire** (`bg-neutral-900 text-white`) : pastille claire
  `#f5f5f5` + texte `#171717` — l'inversion du clair, même geste. `hover:bg-neutral-700`
  glisse vers `#d4d4d4` : le hover « refroidit » vers le milieu, cohérent.
- **Disabled** (`disabled:bg-neutral-300` + `text-white`) : `#171717` sur `#404040`,
  faible comme en clair — exempt WCAG, parité assumée.
- La règle de plancher du design.md §1 se transpose telle quelle : **500 = plancher sur
  carte/page, 600 = plancher sur fond gris (100/200), 300/400 = décoratif seulement.**

## 3. L'accent en sombre

**`#2563eb` ne tient pas sur `#171717` : 3,5:1** — passable pour un glyphe (≥3:1
non-textuel), insuffisant pour du texte. Décision : **`#3b82f6` en sombre** (un cran plus
clair, même famille — 4,9:1 sur carte, 5,2:1 sur page). Le bleu reste immédiatement
reconnaissable comme LE bleu Roadmapped ; sa rareté, elle, ne change pas de régime.

**`accent-tint` sombre : `#1c2636`** = accent sombre mélangé à 14 % dans la carte,
calculé et posé en OPAQUE (même exigence qu'en clair : les arêtes du graphe ne doivent
pas transparaître sous une carte sélectionnée). Le liseré
`shadow-[inset_2px_0_0_var(--color-accent)]` qui l'accompagne partout suit tout seul.
`live-entry-in` (flash d'entrée Activity) part de ce tint : subtil en sombre comme en
clair, c'est le registre voulu.

Cas vérifiés : encre sur tint 14:1 ; accent sur tint 4,9:1 ; badge non-lus `bg-accent
text-white` → texte `#171717` sur `#3b82f6` = 4,9:1 (le badge devient bleu à texte
sombre — correct et net).

## 5. La rampe coolwarm de la température : inchangée

Vérification des trois pôles sur carte sombre `#171717` (exigence non-textuelle ≥3:1,
l'icône est décorative avec valeur texte adjacente) : froid `rgb(59 107 199)` = 3,8:1 ✓ ·
tiède `rgb(163 163 163)` = 7,1:1 ✓ · chaud `rgb(234 88 12)` = 5,0:1 ✓. **Aucun
ajustement** — et un alignement heureux : le pôle tiède `#a3a3a3` coïncide exactement
avec le `neutral-500` sombre, la valeur du texte du badge à côté. « Le tiède se fond
dans le DS » reste vrai dans les deux modes, sans brancher `tempColor()` sur le thème
(pas de plomberie JS de thème — c'est le gain de l'approche tokens, on ne le
compromet pas pour une icône).

Nuance acceptée : l'accent sombre `#3b82f6` se rapproche du pôle froid `#3b6bc7`. Sans
gravité — le thermomètre n'est jamais un signal « actif », il vit collé à du texte mono
gris, et le froid reste plus terne. À réévaluer seulement si une confusion réelle est
observée (feedback sur #269, pas de ticket préventif).

## 6. Ombres : le filet sépare, l'ombre densifiée donne la profondeur

Sur fond sombre, le `shadow-lg` par défaut (noir à 10 %) disparaît. Pas de glow, pas de
bordure ajoutée : **tous les popups/toasts du DS portent DÉJÀ `border border-neutral-200`**
— en sombre ce filet devient `#303030` et assure seul la séparation (y compris popup
sur carte de même valeur, le modèle VS Code). L'ombre est ensuite densifiée pour
retrouver la profondeur, via la couleur d'ombre Tailwind — en ciblant l'élément
porteur (contrainte `@property inherits: false`, cf. Principe) :

```css
/* Ombres en sombre : densifiées (le défaut à 10 % de noir est invisible).
   --tw-shadow-color est @property inherits:false → cibler la classe, pas :root. */
:root[data-theme="dark"] .shadow-sm { --tw-shadow-color: rgb(0 0 0 / 0.45); }
:root[data-theme="dark"] .shadow-lg { --tw-shadow-color: rgb(0 0 0 / 0.6); }
```

`shadow-sm` et `shadow-lg` sont les deux seules ombres nommées du code (vérifié) ; les
ombres arbitraires (`shadow-[inset_2px_0_0_var(--color-accent)]`) suivent leur token.

## 7. Focus, scrollbars, `color-scheme`

- **`:focus-visible`** : `outline: 2px solid var(--color-neutral-900)` — encre dans les
  deux modes (16:1 en sombre), offset 2px conservé.
- **Scrollbars custom** (`.roadmap-cols-scroll`) : `var(--color-neutral-500)` → pouce
  `#a3a3a3` en sombre.
- **Scrollbars natives et widgets système** : poser `color-scheme` — c'est ce qui rend
  sombres les scrollbars non stylées (panneau, popovers, `pre`), les inputs natifs et le
  fond par défaut :

```css
:root { color-scheme: light; }
:root[data-theme="dark"] { color-scheme: dark; }
```

- **`.doc-prose`** : entièrement couvert par la tokenisation §4 — code `#262626` bordé
  `#303030`, blockquote `#b9b9b9` sur filet `#303030`, tableaux et `hr` sur les mêmes
  filets, liens soulignés à l'encre. Rien d'autre à écrire.

## 8. Le bloc CSS complet à ajouter (après le `@theme` de `index.css`)

Dans le `@theme` existant, ajouter une ligne : `--color-page: #fafafa;` (et passer le
`body` sur les `var()`, §4). Puis, en CSS ordinaire :

```css
/*
 * MODE SOMBRE (#269) : le thème n'est qu'un jeu de valeurs — les utilitaires
 * Tailwind lisent var(--color-*), redéfinir les tokens retourne l'app entière.
 * Posé par data-theme sur <html> (script anti-flash dans index.html, état dans
 * src/state/theme.ts). Tri-couche inversé : page #0f0f0f / carte #171717 /
 * filets #303030. Encre #f5f5f5 (jamais #fff — pas de halo). Le cœur de
 * l'échelle réutilise les oklch canoniques en miroir (900↔… zéro dérive).
 */
:root { color-scheme: light; }
:root[data-theme="dark"] {
  color-scheme: dark;

  --color-accent: #3b82f6;        /* #2563eb ne tient que 3,5:1 sur la carte */
  --color-accent-tint: #1c2636;   /* accent 14 % dans la carte, OPAQUE (graphe) */

  --color-white: oklch(0.205 0 0);      /* carte #171717 */
  --color-page: oklch(0.168 0 0);       /* page ≈ #0f0f0f */
  --color-neutral-50: oklch(0.24 0 0);  /* hover rangée, > carte */
  --color-neutral-100: oklch(0.269 0 0);
  --color-neutral-200: oklch(0.31 0 0); /* filets */
  --color-neutral-300: oklch(0.371 0 0);
  --color-neutral-400: oklch(0.556 0 0);
  --color-neutral-500: oklch(0.708 0 0); /* plancher texte : 7,1:1 sur carte */
  --color-neutral-600: oklch(0.78 0 0);
  --color-neutral-700: oklch(0.87 0 0);
  --color-neutral-800: oklch(0.922 0 0);
  --color-neutral-900: oklch(0.97 0 0);  /* encre #f5f5f5 */
}
:root[data-theme="dark"] .shadow-sm { --tw-shadow-color: rgb(0 0 0 / 0.45); }
:root[data-theme="dark"] .shadow-lg { --tw-shadow-color: rgb(0 0 0 / 0.6); }
```

## 9. Le switch — Base UI `Toggle`, icône Trinil, dans le cluster droit du header

**Emplacement** : `ViewHeader`, cluster droit, **entre `<LiveActivityMenu />` et
`{children}`** — le chrome d'app (MAJ · Activity · thème) reste contigu, les contrôles
de vue (filtres, recherche, « + task ») gardent le bord droit. Ordre final :
`UpdateNotice · LiveActivityMenu · ThemeToggle · children`.

**Composant** : Base UI **`Toggle`** (`@base-ui/react/toggle`) — un bouton à état
pressé, pas un `Switch` à glissière : le DS du header parle en boutons-icônes bordés,
une glissière serait le seul élément de ce langage dans l'app. Icônes **Trinil `Sun` /
`Moon`** (exports vérifiés dans `trinil-react`). Convention : l'icône montre la
DESTINATION — `Moon` en clair (« passe en sombre »), `Sun` en sombre — avec l'`aria-label`
assorti. Fichier `src/components/ThemeToggle.tsx` :

```tsx
const dark = theme === 'dark'
<Toggle
  pressed={dark}
  onPressedChange={(p) => setTheme(p ? 'dark' : 'light')}
  aria-label={dark ? 'Switch to light theme' : 'Switch to dark theme'}
  title={dark ? 'Light theme' : 'Dark theme'}
  className="flex items-center rounded-md border border-neutral-300 bg-white px-2 py-1
             text-neutral-600 transition-colors hover:bg-neutral-100"
>
  {dark ? <Sun size={12} className="my-0.5" /> : <Moon size={12} className="my-0.5" />}
</Toggle>
```

(`my-0.5` donne à l'icône la boîte de 16px d'une ligne `text-xs` → même hauteur exacte
que les triggers `FilterMenu` voisins, `py-1` identique.)

**État — `src/state/theme.ts`, PAS de provider.** `ViewHeader` est monté 4× et remonté à
chaque changement de vue (contrainte documentée dans la spec live-updates-v2) — mais ici
la source de vérité est `document.documentElement.dataset.theme` + `localStorage`, qui
survivent à tout remount : un hook local suffit, aucun contexte.

```ts
export type Theme = 'light' | 'dark'
const KEY = 'ui:theme'   // convention existante nav:view, graph:epics…

export function setTheme(t: Theme)        // dataset.theme + localStorage (try/catch)
export function useTheme(): [Theme, (t: Theme) => void]
// - init : lire dataset.theme (posé par le script anti-flash — source unique au boot)
// - resolveTheme(stored, systemDark) : pure, exportée pour le test
// - tant qu'AUCUN choix explicite n'est stocké : écouter
//   matchMedia('(prefers-color-scheme: dark)') 'change' et suivre le système
// - au premier clic : le choix devient explicite (stocké), l'écoute système cesse
```

Défaut = `prefers-color-scheme` ; le clic fige un choix explicite. Pas de troisième état
« system » dans l'UI (ponytail) — consigner en dette :
`quick "Theme toggle : pas de retour explicite au mode système après un choix manuel" --tags debt`.
Pas de synchronisation inter-onglets (`storage` event) : chaque dashboard multi-repos vit
sur son propre port, donc sa propre origine localStorage — il n'y a rien à synchroniser.

**Anti-flash — script inline dans `index.html`**, dans `<head>` juste après le
`<meta viewport>` (avant tout CSS/JS, exécuté avant le premier paint — sans lui,
un rechargement en sombre flashe blanc) :

```html
<script>
  // Anti-flash thème (#269) : pose data-theme AVANT le premier paint.
  // Doit rester inline et synchrone. Miroir de la résolution de src/state/theme.ts.
  (function () {
    var t = null
    try { t = localStorage.getItem('ui:theme') } catch (e) {}
    if (t !== 'dark' && t !== 'light')
      t = window.matchMedia && matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light'
    document.documentElement.dataset.theme = t
  })()
</script>
```

## 10. Cas limites & garde-fous

- **Aucun `dark:` variant, aucun état de thème dans les composants métier** — si un
  composant a besoin de « savoir » le thème, c'est un défaut de tokenisation à corriger
  côté tokens.
- **Anims** : `pulse-live` (opacité seule) et `.chev` sont theme-agnostiques ;
  `live-entry-in` suit `--color-accent-tint` automatiquement. Rien à faire.
- **Transition de bascule** : ne PAS ajouter de transition globale sur le changement de
  thème (vague de couleurs qui rampent) ; les `transition-colors` locaux existants
  suffisent.
- **Build démo / homepage** : le bundle démo est hébergé par `5e1y/roadmapped-site`
  (Cloudflare Pages) — le script anti-flash doit être répliqué dans la page HTML de CE
  repo-là, sinon la démo flashe. À signaler à la livraison, pas dans ce repo.
- **Copy de la démo** (`src/demo/tree.ts` ligne ~109) : « no toggle to maintain, the OS
  already has one » devient faux. Remplacer par, au ton Rémi :
  `"One accent color, used rarely enough to mean something. Dark mode follows your OS — and there's a toggle for the contrarians."`
- **`docs/design.md` §1** : ajouter la colonne sombre au tableau des tokens + la règle
  « le thème est un jeu de valeurs, hex en dur interdit » dans la même livraison.
- **Contrastes vérifiés** (à re-vérifier à l'œil, pas à recalculer) : encre/carte 16:1 ·
  500/carte 7,1:1 · 600/`neutral-100` 7,7:1 · accent/carte 4,9:1 · accent/page 5,2:1 ·
  `text-white` sur accent 4,9:1 · sur `neutral-900` 16:1 · 400/carte 3,8:1 (décoratif
  seulement, comme en clair).
