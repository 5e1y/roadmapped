# Spec — Notepad : incubateur d'idées et créateur de prompts

**Date** : 2026-07-07 · **Statut** : DRAFT — en attente d'approbation Rémi
**Brainstorm** : 3 questions tranchées par Rémi le 2026-07-07 (stockage, périmètre v1, cycle de vie).

## Contexte

Pendant que l'agent travaille, les idées arrivent en rafales — elles finissent dans la
barre du terminal (polluée tant que le prompt n'est pas mûr) ou hors de l'outil. Le
Notepad est le sas : noter vite, affiner, puis servir un prompt propre à l'agent.

## Décisions (et alternatives écartées)

1. **Stockage : `docs/notes/`, GITIGNORÉ** (décision Rémi). Notes purement locales,
   jamais dans l'historique — zéro bruit dans les commits. *Écartés* : commité
   (bruit d'historique pour des brouillons), visible dans Docs (mélange brouillon
   perso / connaissance projet). Conséquences : l'app crée `docs/notes/` au premier
   usage ET ajoute la ligne à `.gitignore` si absente ; `buildDocsTree` exclut
   explicitement `notes/` (comme `node_modules`) ; AVERTISSEMENT affiché une fois
   dans l'UI : « notes locales, non versionnées, non sauvegardées par git ».
2. **Périmètre v1** (décision Rémi) : socle (liste gauche + éditeur autosave +
   Cmd+N + copier) + **pièces jointes en liens** + **« Copier pour l'agent »**.
   *Reportés v2* : templates de prompts, pont « → ticket ».
3. **Cycle de vie : auto-archive à la conversion** (décision Rémi). S'appliquera
   quand le pont « → ticket » arrivera (v2) ; en v1, bouton d'archivage MANUEL
   (déplace dans `notes/_archive/`) + suppression.

## Design

- **4e tab « Notepad »** dans le header (après Docs). Layout : flanc gauche 420px
  (gabarit Docs/radar) = liste des notes triées par modification décroissante
  (nom + date, bouton « + note » = Cmd+N) ; zone principale = éditeur.
- **Éditeur écriture-d'abord** : textarea ghost plein écran (même peau que le détail
  de tâche), autosave au blur + debounce 800ms, indicateur « enregistré » discret.
  Titre = première ligne de la note (fichier `<slug>.md`, renommé au fil de l'eau).
  Raccourcis : Cmd+N (nouvelle note), Cmd+S (save forcé), Cmd+Shift+C (copier pour
  l'agent). Compteur de caractères ≈tokens (chars/4) en pied — on écrit des prompts,
  autant voir leur coût.
- **Pièces jointes en liens** : drag & drop d'un fichier → insère son CHEMIN ABSOLU
  en ligne dédiée (`[fichier: /Users/…/maquette.png]`) — JAMAIS de copie. Clic sur la
  ligne → `POST /api/reveal` qui exécute `open -R <path>` (macOS ; `xdg-open` du
  dossier parent sur Linux, `explorer /select` sur Windows). Sécurité : endpoint
  local-only (comme le reste de l'API), chemin absolu exigé, existence vérifiée,
  refus des chemins hors du HOME de l'utilisateur.
- **« Copier pour l'agent »** : copie la note nettoyée (lignes `[fichier:…]`
  converties en chemins nus, titre en première ligne) — prête à coller dans le
  terminal. Le copier brut (Cmd+C classique) reste évidemment possible.
- **Token-neutre** : rien dans le skill, rien dans les refs par défaut — l'agent ne
  lit une note que si Rémi la lui colle.

## API

`GET /api/notes` (liste + dates), `GET/PUT /api/notes/:slug` (contenu),
`POST /api/notes` (création), `POST /api/notes/:slug/archive`, `DELETE /api/notes/:slug`,
`POST /api/reveal` (chemin absolu → Finder). Fichiers plats, pas de front-matter :
le nom de fichier et les stats du FS suffisent.

## Hors périmètre (v2 explicitement)

Templates de prompts (`notes/_templates/`) ; pont « → ticket » (CreateTaskPanel
pré-rempli + auto-archive) ; recherche dans les notes ; sync/multi-machines.

## Critères de fini

1. Créer/éditer/archiver une note au clavier sans toucher la souris (Cmd+N, taper,
   blur) ; autosave vérifié par relecture du fichier sur disque.
2. Drag & drop d'un fichier → lien inséré, clic → le Finder s'ouvre sur le fichier
   (vérifié sur macOS) ; chemin hors HOME refusé (test API).
3. « Copier pour l'agent » produit le prompt nettoyé (asserté sur le presse-papier).
4. `docs/notes/` gitignoré (git status propre après création de notes), invisible
   dans l'onglet Docs.
5. Tests + build verts.
