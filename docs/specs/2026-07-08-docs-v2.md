# Vue Docs V2 — tri, regroupement, recherche, épinglage, backlinks

**Date** : 2026-07-08 · **Statut** : DRAFT — en attente de relecture Rémi
**Prédécesseur** : `2026-07-07-roadmaped-v2-design.md` §« Vue Docs (phase 3 — lecteur seul) »

## Problème

La Vue Docs livrée en phase 3 est un lecteur nu : `buildDocsTree` (`src/server/docs.ts`)
rend l'arbre physique de `docsDir`, trié dossiers-puis-fichiers en ordre naturel, et
`DocsView.tsx` affiche le markdown. C'est tout. Concrètement, avec le corpus actuel
(`find docs -name '*.md'`), on a d'un côté un dossier `specs/` qui gonfle (9 fichiers,
un par jour), de l'autre une dizaine de `.md` en vrac à la racine — `audit-a11y-*`,
`audit-ux-*`, `guide.md`, `design.md`, `tone-of-voice.md`, `comms-plan.md`,
`site-copy.md`… mélangés sans logique de lecture.

Frictions constatées dans le code :
- **Pas de tri utile.** L'ordre alpha-numérique est le seul disponible. Pour retrouver
  « ce que j'ai touché hier », il faut scanner à l'œil. `DocNode` ne porte même pas de
  date de modification : seul `createdAt` (birthtime, `docs.ts:59`) existe, et il n'est
  affiché que sur les feuilles.
- **Pas de regroupement.** L'arbre suit le disque : les audits et les guides,
  éparpillés à la racine, ne se regroupent nulle part.
- **Pas de recherche.** Aucun champ pour filtrer l'arbre ; on déplie à la main.
- **Aucun lien doc ↔ tâche.** Un doc qui écrit « voir #114 » ne pointe vers rien.
  Le champ `links: number[]` existe côté tâche (`tasks.ts:67`) mais reste inexploité
  côté docs. La v2-design classait les wikilinks en « différés » — on les ouvre ici.

## Décision (tranchée par Rémi — rappel)

Périmètre **riche**, cinq briques :

1. **Tri** de l'arbre : *récemment modifié* ⇆ *alphabétique*, togglable.
2. **Regroupement logique** : *specs*, *guides*, *audits* (et *autres*), en plus de la
   vue par dossiers physiques.
3. **Recherche / filtre** dans l'arbre (champ de saisie, filtrage par nom).
4. **Date de modification visible**, en relatif, via `relativeTime` (`src/lib/relativeTime.ts`, #126).
5. **Épinglage** des docs importants, persisté.
6. **Backlinks doc ↔ tâches** : les `#id` mentionnés dans un doc remontent vers les
   tâches réelles (et réciproquement, la tâche liste les docs qui la citent).

Ces points sont actés, pas à ré-ouvrir. On reste dans la doctrine maison : source de
vérité = fichiers sur disque, lecture seule côté docs, monochrome, état d'UI persisté
via `uiPersist`.

## Conception

### Vue d'ensemble

Le serveur reste fin : il enrichit `DocNode` (date de modif), extrait les refs `#id`, et
expose un index de backlinks corpus. Toute l'**organisation** (tri, groupes, filtre,
ordre des épinglés) est **calculée côté client**, à partir de l'arbre chargé une fois —
c'est de la présentation, ça n'a pas à toucher le disque, et ça reste testable en pur.

### 1. Date de modification — serveur (`src/server/docs.ts`)

`DocNode` gagne `modifiedAt?: string` (AAAA-MM-JJ), rempli dans `buildDocsTree` depuis
`st.mtime` — pendant du `createdAt` (birthtime) déjà présent ligne 59. On garde
`createdAt` (utile pour départager, et déjà affiché). L'affichage bascule sur
`modifiedAt` : « récemment modifié » veut dire *touché*, pas *créé*.

`DocsTreeFile` (`DocsTree.tsx:87`) affiche déjà `relativeTime(node.createdAt)` avec
`absoluteDate` en `title` — on remplace la source par `modifiedAt ?? createdAt`. Zéro
nouveau composant de date.

### 2. Extraction des refs `#id` — serveur (`src/server/docs.ts`)

Fonction **pure** `extractTaskRefs(content: string): number[]` : match global `/#(\d+)\b/`,
dédupliqué. Pure = testable sans disque (même doctrine que `refExtract.ts`).

Le bruit (`#fafafa`, `#e5e5e5` de `design.md`, ancres markdown `#titre`) est écarté
**par construction** : un id de tâche est un entier, une couleur hex contient des
lettres, une ancre est un mot. Deuxième garde décisive : on ne retient que les ids qui
**existent réellement** dans l'arbre de tâches — le recoupement se fait au moment de
servir, pas dans la fonction pure.

`getDocContent` (`api.ts:161`) répond aujourd'hui `{ ok, content }`. On ajoute
`taskRefs: number[]` : dans `runAction`, on a `tasksDir` sous la main → on charge les ids
existants (via `treeWithErrors` + `findTaskInTree`, ou un helper `allTaskIds(tree)`) et on
filtre `extractTaskRefs(content)` contre eux. Une tâche archivée compte (elle existe dans
`tree.archive`, `findTaskInTree` la voit déjà).

### 3. Index backlinks corpus — serveur (`src/server/docs.ts` + `api.ts`)

Pour le sens inverse (« quels docs me citent ? », affiché dans le side panel tâche), on
scanne tout le corpus une fois : `buildBacklinkIndex(docsDir): Record<number, string[]>`
= pour chaque id de tâche, la liste des chemins de docs qui le mentionnent. Réutilise
`extractTaskRefs` sur chaque `.md` (marche récursive, mêmes exclusions que
`buildDocsTree` : `.notes`, `node_modules`, cachés).

Nouvelle action `getDocsBacklinks` → `GET /api/docs/backlinks`, payload
`{ ok: true, backlinks: Record<number, string[]> }`. Recalculé à chaque requête (corpus
petit, pas de cache — cohérent avec « pas d'état dérivé stocké »).

### 4. Organisation client — `src/lib/docsOrganize.ts` (nouveau, pur)

Toute la logique de présentation, testable sans React :

- **`sortTree(nodes, mode)`** — `mode ∈ 'alpha' | 'recent'`. Récursif. `alpha` =
  comportement serveur actuel (dossiers d'abord, `localeCompare` numérique). `recent` =
  tri sur `modifiedAt` desc (les dossiers portent le `modifiedAt` max de leurs feuilles,
  calculé à la volée).
- **`filterTree(nodes, query)`** — garde toute feuille dont le nom contient `query`
  (insensible casse/accents) **et** les dossiers ancêtres nécessaires. Query vide =
  arbre intact. Recherche par **nom** uniquement (le plein-texte reste la « recherche
  globale » différée en V3 par la v2-design).
- **`groupByCategory(nodes)`** — aplatit les feuilles et les range en buckets ordonnés :
  `specs` (chemin sous `specs/`), `audits` (nom `audit-*`), `guides`
  (`guide|design|tone-of-voice|site-copy|comms-plan|process-*`), `autres` (le reste).
  La règle de catégorisation vit dans un seul `categorize(path): Category` documenté.
- **`applyPins(nodes, pinned)`** — extrait un groupe *Épinglés* en tête, dans l'ordre du
  tri courant.

L'ordre d'application : `filter` → (`groupByCategory` si mode groupes) → `sort` dans
chaque groupe → `applyPins`. `DocsTree.tsx` reste bête : il reçoit des `DocNode[]`
(éventuellement des pseudo-dossiers de groupe) et rend récursivement comme aujourd'hui.
Les groupes logiques et *Épinglés* sont modélisés comme des `DocNode` sans `modifiedAt`
avec `children` → réutilise `DocsTreeFolder` tel quel (Collapsible), zéro branche neuve.

### 5. Où vit l'état

- **Préférences persistées** via `uiPersist.ts` (`usePersistentStrings`, déjà le canal
  des accordéons) :
  - `docs:sort` → `['recent']` | `['alpha']` (défaut `recent`)
  - `docs:group` → `['folders']` | `['categories']` (défaut `folders`)
  - `docs:pinned` → liste de chemins repo-relatifs (`node.path`)
  Un petit hook `useDocsPrefs()` enveloppe ces trois clés (lecture + setters).
- **Query de recherche** : `useState` local dans `DocsView` — c'est un filtre éphémère,
  pas une préférence, il ne survit pas au reload (volontaire).
- **`taskRefs` du doc courant** : dans l'état de `DocsView`, à côté de `content`
  (même `fetch`, même cycle).
- **Backlinks corpus** : chargé par un hook `useDocsBacklinks()` calqué sur
  `useDocsTree()` (fetch une fois, lecture seule), consommé par le side panel tâche.

### 6. UI

**Barre d'outils du flanc gauche** (`DocsView.tsx`, au-dessus de l'arbre, à la place / à
côté du libellé « Fichiers » ligne 103) :
- champ de recherche (input permanent camouflé — cf. doctrine ghost-inputs), icône loupe ;
- deux toggles compacts : tri (récent / A-Z) et regroupement (dossiers / catégories),
  en boutons segmentés monochromes.

**Ligne de fichier** (`DocsTreeFile`) : ajout d'un bouton épingle (survol/actif), à
gauche ou à droite du nom, qui bascule `docs:pinned`. Le pin ne réordonne pas dans le
dossier : les épinglés apparaissent **en plus** dans le groupe *Épinglés* en tête
(l'entrée d'origine reste à sa place — voir points ouverts).

**Panneau « Tâches liées »** dans `DocsView` (sous le titre du doc rendu, ou en encart
en tête de la zone de lecture) : liste des `taskRefs` (code + titre, résolus via l'arbre
de tâches chargé). Clic → navigation cross-vue : `setView('backlog')`
(`ViewContext`) + `openTask(id)` (`PanelContext`, `api.ts` PanelState:29). Les deux
existent déjà, il suffit de les câbler.

**Side panel tâche — « Mentionné dans »** : nouvelle section listant
`backlinks[task.id]` (chemins de docs). Clic → `setView('docs')` + `onSelectDoc(path)`.
C'est le pendant exact du panneau doc→tâche.

## Découpage en tâches d'implémentation

Ordonnées, chaînables par `dependsOn`. Chaque tâche = un lot testable.

1. **`docs.ts` — `modifiedAt` sur `DocNode`.** Ajout du champ + remplissage `st.mtime`
   dans `buildDocsTree`. Tests unitaires (fixture avec mtimes contrôlés).
   → *dependsOn: []*
2. **`docs.ts` — `extractTaskRefs` (pur) + `taskRefs` dans `getDocContent`.** Fonction
   pure testée (hex/ancres exclus), câblage dans `runAction` avec filtrage contre les ids
   réels (helper `allTaskIds`). Tests unitaires + intégration (doc citant #114 et #ffffff).
   → *dependsOn: []*
3. **`docs.ts` + `api.ts` — index backlinks corpus + `GET /api/docs/backlinks`.**
   `buildBacklinkIndex`, action `getDocsBacklinks`, route. Tests d'intégration sur un
   `docsDir` temporaire.
   → *dependsOn: [2]* (réutilise `extractTaskRefs`)
4. **`src/lib/docsOrganize.ts` — logique pure d'organisation.** `sortTree`,
   `filterTree`, `groupByCategory`/`categorize`, `applyPins`. Tests unitaires exhaustifs
   (c'est le cœur testable de la feature).
   → *dependsOn: [1]* (le tri `recent` a besoin de `modifiedAt`)
5. **État — `useDocsPrefs()` (clés `docs:sort` / `docs:group` / `docs:pinned`).**
   Hook au-dessus de `uiPersist`, plus `useDocsBacklinks()` calqué sur `useDocsTree`.
   → *dependsOn: [3]*
6. **UI arbre — barre d'outils + épingles + date de modif.** Champ recherche, toggles
   tri/groupe, bouton épingle par ligne, bascule affichage `modifiedAt`. Câblage
   `DocsView` → `docsOrganize` → `DocsTree`.
   → *dependsOn: [4, 5]*
7. **UI — panneau « Tâches liées » (doc → tâche) + nav cross-vue.** Rend `taskRefs`,
   clic ouvre la tâche (`setView` + `openTask`).
   → *dependsOn: [2, 5]*
8. **UI — side panel tâche « Mentionné dans » (tâche → docs).** Consomme
   `useDocsBacklinks`, clic ouvre le doc (`setView('docs')` + sélection).
   → *dependsOn: [5]*

## Risques / points ouverts

- **`mtime` ≠ vraie date d'édition.** Après un `git clone` frais, tous les fichiers
  partagent la mtime du checkout → le tri « récemment modifié » devient plat. La
  v2-design proscrit `git` dans `docs.ts` ; on assume `mtime` (birthtime en repli, déjà
  le cas). Si insuffisant en pratique, une lecture `git log -1` en fallback serait un
  ajout ciblé, hors périmètre ici.
- **`categorize` — taxonomie des fichiers racine.** Le mapping proposé
  (`audit-*` → audits, `guide|design|tone-of-voice|site-copy|comms-plan|process-*` →
  guides) est une heuristique par nom. À valider / compléter par Rémi : quels fichiers
  vont dans *guides* vs *autres*, faut-il une 4ᵉ catégorie ?
- **Épinglés : doublon ou déplacement ?** Proposé : l'épinglé apparaît dans le groupe
  *Épinglés* **et** reste à sa place d'origine. L'alternative (le retirer de son dossier)
  évite la redite mais casse le repère spatial. Choix UI à trancher.
- **Refs `#id` dans des blocs de code / URLs.** `extractTaskRefs` ne comprend pas le
  markdown ; un `#42` dans un fence ` ``` ` serait capté. Le recoupement avec les ids
  réels limite les faux positifs mais ne les annule pas. Acceptable pour un lecteur
  local ; à revoir si le bruit se manifeste.
- **Portée de la recherche = noms.** Le plein-texte reste la « recherche globale »
  différée en V3 (v2-design §Hors périmètre). On ne l'ouvre pas ici.

> **Décision verrouillée (Rémi, 2026-07-08)** : un doc épinglé apparaît **en doublon** — dans
> un groupe « Épinglés » en tête ET à sa place d'origine dans son dossier (on le retrouve aux
> deux endroits). La taxonomie `categorize()` reste à l'heuristique proposée (audit-* →
> audits ; guide/design/tone-of-voice/site-copy/comms-plan/process-* → guides ; sinon autres),
> ajustable à l'usage.
